The rolling log records every tool call: a line per operation, written by the PostToolUse
hook. The tempting design is a small database that keeps running totals. The harness refuses
it, for one reason that decides everything else.

<Fact>Two PostToolUse hooks can run concurrently. The only write that is safe under that race is an append.</Fact>

A read-modify-write needs the file to hold still between the read and the write. Under two
racing processes it will not, and one update silently clobbers the other. An append has no
such window: each process opens, writes its line, and closes. Nothing is read, so nothing
can go stale. The cost is that any number you compute later, like how many times a file was
touched, is a floor, not a guarantee. For advisory memory, a slightly low count is fine; a
corrupted log is not.

<EvidencePin file="reference/hooks/unified/modules/rolling-log.mjs">
The log appends one record per operation and never rewrites the file. Counters derived from
it are explicitly treated as approximate, which is the price of being safe under concurrency.
</EvidencePin>

This is the first and cheapest tier of memory. The next room shows the other three, and
which one actually costs you anything.
