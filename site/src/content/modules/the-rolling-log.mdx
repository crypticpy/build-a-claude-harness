import EvidencePin from '../../components/heva/EvidencePin.astro';

The rolling log records every tool call: the PostToolUse hook writes one line per operation to
a per-session file. Beside it the harness keeps a second structure, a per-file edit counter in
file-edits.json, and the two are written in deliberately different ways.

<Fact>Two PostToolUse hooks can run at once. The harness picks a durability guarantee for each record by how much it would cost to lose it.</Fact>

The operation record is the one you cannot lose, so it is appended. Each process opens the
session log, writes its line, and closes. Nothing is read first, so two racing hooks cannot
clobber each other, and no line is ever lost.

The per-file counter is different. To bump it, the harness reads file-edits.json, increments
the stored count, and writes the whole file back. That is a read-modify-write, the very pattern
an append avoids, and nothing locks the file to hold it still in between. So two hooks can read
the same count and both write back, and one increment is lost, last writer wins. The harness
accepts that on purpose, because the counter is advisory. A tally that is occasionally one low
costs nothing; a lost operation record would.

The same instinct shows up once more. The optional background summary of each edit is appended
to its own sidecar file, precisely so a slow write can never race the counter it would otherwise
share.

<EvidencePin file="reference/hooks/unified/modules/rolling-log.mjs">
Two records, two write paths. The per-operation log is appended one line per tool call, which
racing processes cannot clobber. The per-file counter is read, incremented, and written back with
no lock in between, so two concurrent hooks can lose an increment, last writer wins.
</EvidencePin>

This is the first and cheapest tier of memory, and it already carries the whole principle: protect
a value as hard as it costs to lose it, and no harder. The next room shows the other three tiers,
and which one actually costs you anything.
