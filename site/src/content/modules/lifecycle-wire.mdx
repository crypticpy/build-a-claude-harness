A Claude Code session is not a loop you write. It is a sequence of **platform events**,
and the harness gets to subscribe to the ones it cares about. Each subscription is a row
in a settings file: an event name, a matcher that decides which tool calls it applies to,
and a command to run. When the event fires, the platform runs the matching commands and
reads what they print.

So the whole lifecycle reduces to one question you can answer by reading a file: which
events did this harness wire, and what runs at each one?

<Fact>This harness wires five of the six session lifecycle events. PreToolUse is left unsubscribed.</Fact>

Five of the six events carry a single hook. The sixth, PostToolUse, carries **two**, and
the order matters: the formatter runs first on file edits, then a catch-all logger runs on
every tool. That ordering is not incidental. The formatter has to touch the file before the
logger records that the turn happened.

<EvidencePin file="reference/settings.template.json">
Every subscribed event in the visualization below is a real row in the settings file the
build reads at the pinned commit. If a row changes, the model regenerates and the build
fails until the picture and the source agree again.
</EvidencePin>

The one event with no row is PreToolUse. It is a real platform event, fired before every
tool call, but this harness publishes no hook for it. There is nothing to draw except its
absence, so the diagram keeps it as a dashed, hollow station: present in the lifecycle, but
not wired here.

<Caveat>The event set belongs to the platform, not to this harness. A future Claude Code release could add, rename, or remove an event. The rows pinned here are real as of the pinned commit, and no further.</Caveat>
