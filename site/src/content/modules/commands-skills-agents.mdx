Three things in the harness look similar from the outside, because all three end up as work
the model does. What separates them is who pulled the trigger.

A slash command is yours. You type `/evolve`, and it runs. A skill is the model's call: a
prompt comes in, the activation hook checks it against a rule set, and if it matches, the
skill is offered. A subagent is neither, exactly; it is a fresh context spun up to do one
bounded job and report back, so its work never crowds your main thread.

<Fact>Commands are invoked by you. Skills are suggested by the model. Subagents run in a context of their own.</Fact>

The skills part has a trap. There are two files named `skill-rules.json`, and only one is
real. The activation module reads the copy under `hooks/unified/`, and the root-level file
is a stub that redirects you there. Edit the decoy and your rule never fires, silently,
because a fail-silent harness will not tell you that you edited the wrong file.

<EvidencePin file="reference/hooks/unified/modules/skill-activation.mjs">
The activation module names its canonical rules path explicitly. The root-level file is not
the one it loads, which is exactly the kind of thing fail-silent will never warn you about.
</EvidencePin>

Subagents are the most interesting of the three, because running several at once has a rule
of its own. That is <CrossPhaseLink to="phase-1/subagents-fan-out">the next room</CrossPhaseLink>.
