This is the climax of the descent, and the one place autonomy gets dangerous. If an agent can
act without you, something has to decide which actions are fine to take and which are not.
The gate is that decision, made deterministic so you can watch it work.

<Fact>The gate lets safe actions glide through and arrests the named harms. Allow is friction-free; Deny is where it stops.</Fact>

Read the action, check it against a short list of named harms, and either pass it or stop it
at the diamond. Editing a file in the project: pass. Running the tests: pass. Posting a
credentials file to the network, deleting outside the working directory, force-pushing a
protected branch: each one trips a named rule and is arrested, with a human glyph marking the
point where you would be asked to intervene.

<EvidencePin file="course/07-level-up/7.2-cf-approve/solution/evaluate-policy.mjs">
The policy evaluates a proposed action against the named harms and returns allow or deny.
The deny set is explicit and small, which is what makes the gate's judgement legible.
</EvidencePin>

Try the judgement yourself. Sort each request into the bin the gate would choose, then check
your reasoning against the discriminating feature for each one.

<ScenarioSorter />

<Caveat>This gate is a friction tool, not a security boundary. It is a deterministic stand-in for the real second-LLM check, course-taught rather than a wired harness layer. It catches the obvious and the accidental; it is not a guarantee against a determined actor, and must never be trusted as the last line of defense.</Caveat>
