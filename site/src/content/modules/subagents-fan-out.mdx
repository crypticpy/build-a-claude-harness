Fanning out to several subagents at once is how you turn a long serial job into a short
parallel one. The speed-up is real, but it has a precondition that is easy to skip past in
the rush to go faster.

<Fact>Parallel agents must own disjoint files. If two would touch the same file, you sequence them instead.</Fact>

Nothing in the harness locks a file while one agent edits it. So two agents writing the same
file are simply racing, and the loser's work disappears. The fix is not a lock; it is the
work assignment. Give each agent a set of files no other agent will touch, and the parallel
run is safe by construction. When the task cannot be carved up that cleanly, the honest move
is to run the steps in order and accept the slower clock.

<EvidencePin file="reference/commands/plan.md" lines={[52]}>
The plan command states the rule directly: assign each parallel sub-agent a file set it owns
exclusively, and if the steps share any file, run them sequentially instead of fanning out.
</EvidencePin>

Watch the visualization with the overlap toggle on. The moment two lanes claim the same
file, the parallel option is struck out, and the only legal path left is one lane at a time.
